# 路由算法模式

本文描述 AIHubRouter 当前实现的路由规则。它用于解释“推荐”“模拟”和“立即路由”中的选择依据；三者共享同一套决策引擎。程序只修改已选 API Key 的 `group_id`，不修改模型名称或调用请求。

## 1. 计算输入

每轮路由先读取供应商、账户可用分组、用户专属倍率和 API Key 状态。候选分组必须同时满足：

- 分组为 `active`，且平台与当前平台一致。
- 供应商最新的 `Enabled` 和 `Available` 状态为真。
- `GroupId` 有效，公开倍率或用户专属倍率为有限且非负数。
- 当前账户有该分组的授权。
- 不在分组 ID 或节点名称黑名单中。
- 归一化到 `0..1` 的 6 小时成功率达到用户设置的最低值。
- 没有仍在有效期内的主动健康检查失败。
- 至少存在一个可用证据：`CheckedAt`、仍在有效期内的主动检查成功、有限且大于 0 的首 Token 延迟，或有限且大于 0 的输出 Token 速度。

同一分组存在多条供应商记录时，路由层只保留一个代表记录。有效倍率 `EffectiveMultiplier` 的优先级是：用户专属倍率覆盖值优先，否则使用公开倍率。

证据超过 30 分钟不会使节点自动失效。证据权重在前 30 分钟（含边界）为 `1`，之后为 `max(0.25, 30 分钟 / 证据年龄)`；只有有效性能值但没有时间戳时权重为 `0.25`。该权重只削弱正向速度和可靠性收益，价格惩罚、速度变慢、可靠性下降等负面因素保持原值。最新禁用/不可用、未授权、命中黑名单、倍率无效、成功率不达标和新鲜主动失败始终是硬门槛，历史数据不能把它们重新变为可路由。

只有首 Token 而没有输出 Token 速度时，节点仍可参与可用性、价格和首字延迟排序，但不能凭空满足需要完整生成时间的均衡 Deadline 或速度切换条件。

### 30 分钟滚动指标

程序在进程内按 `平台 + GroupId` 保留最近 30 分钟的观测值。公开倍率、用户专属倍率、首 Token 延迟、输出速度、最后调用时间和 6 小时成功率等数值使用 P50/中位数；保守规则另取首 Token 延迟 P90 和输出速度 P25。`Enabled`、`Available` 和 `CheckedAt` 始终取最新代表记录的状态，绝不对历史布尔值或检测时间求中位数。

窗口会在程序退出后自然重建；更换站点地址、邮箱、密码、Token、Cookie 或 User-Agent 时也会清空，避免把不同账户或站点的数据混在一起。

### Key 健康检查

程序可选地检查一把用户指定的 Key 是否仍能调用。用户配置该 Key 的 ID、对应的 API Key 和一个可用测试模型后，程序按配置周期（默认 90 秒）读取该 Key 当前的 `GroupId`，只向当前分组发送一次 `/v1/chat/completions` 流式 `ping` 请求（`stream=true`、`max_tokens=1`）。收到第一个有内容的 SSE token 即判定为存活，并在状态栏显示首 Token 延迟。

定时健康检查不遍历节点，也不切换这把 Key 的分组；启用后该 Key 会从普通业务路由中排除。成功和失败都会按精确的 `平台 + GroupId` 写入路由证据：成功可提供新鲜时间戳和首 Token 延迟，失败在有效期内是硬门槛。有效期为 `min(2 * 检查周期, 30 分钟)`，边界时刻仍有效；默认 90 秒周期对应 180 秒。过期成功和失败都恢复为中性，不再替代供应商证据或阻止节点。

## 2. 先筛选，再排序

所有模式都先执行上面的硬性筛选。没有合格分组时，不会写入 Key。

普通推荐分数使用“延迟收益减去价格溢价，再加可靠性差异”的形式：

```text
speedup = baselineLatency / candidateLatency - 1
premium = (candidateMultiplier - minimumMultiplier) / minimumMultiplier
reliabilityDelta = normalize(candidate.SuccessRate6h) - normalize(baseline.SuccessRate6h)

weightedSpeedup = speedup > 0 ? speedup * evidenceWeight : speedup
weightedPositiveReliabilityDelta = reliabilityDelta > 0
    ? reliabilityDelta * evidenceWeight
    : reliabilityDelta

score = latencyWeight * weightedSpeedup
        - priceWeight * premium
        + 0.15 * weightedPositiveReliabilityDelta
```

`SuccessRate6h` 在计算前归一化到 `0..1`；缺失或非有限值按 `0` 处理。权重只用于得到推荐基线和部分候选排序；真正是否切换仍受后续模式规则约束。

| 用户模式 | 价格权重 | 延迟权重 |
| --- | ---: | ---: |
| 经济 | 95% | 5% |
| 均衡 | 80% | 20% |
| 速度 | 35% | 65% |

缺失或无效的首 Token 延迟不会冒充为快速节点；如果没有可用延迟数据，系统回退到最低有效倍率。

### 强制分组

在供应商表格中右击一个可路由节点，可选择“强制使用分组，直到不可用”。强制状态按 `GroupId` 写入 `route-state.json`，跨程序重启保留；再次右击同一分组可手动取消。

强制分组通过显式优先级覆盖经济、均衡和速度决策，并绕过策略切换防抖。程序不会修改该节点的倍率、参考分或性能观测；界面会在算法排行列显示“固定”、决策列显示“强制”。当目标不再满足任一硬性筛选条件时，拿到完整账户数据后的预览、模拟或真实路由都会自动解除强制状态，再按当前模式选择可用节点；解除时清空候选观察计数，但保留已经建立的 Policy 驻留基线。模拟本身仍不会修改任何 Key。

## 3. 调用间隔如何覆盖模式

“用户模式”是界面选择的经济、均衡或速度。经济始终解析为严格经济，均衡始终解析为 Deadline；只有用户选择速度时，“有效偏好”才按当前分组的最后调用结束时间动态覆盖：

| 速度模式下的调用间隔 | 有效偏好 |
| --- | --- |
| 小于 5 秒 | 速度 |
| 5 到 15 秒（含边界） | 速度 |
| 大于 15 到 30 秒（含 30 秒） | 均衡 |
| 大于 30 秒 | 经济 |
| 没有有效最后调用时间 | 速度 |

因此，短调用间隔不会把用户选择的经济升级为速度，也不会把用户选择的均衡改成其他策略。

## 4. 经济模式

经济模式，以及被调用间隔覆盖为经济的模式，选择符合硬性筛选条件的严格最低有效倍率分组。

同倍率时依次比较：6 小时成功率、首 Token 延迟、分组 ID。它不为了小幅速度收益支付更高倍率；当前分组按倍率、成功率、首 Token 延迟和分组 ID 的完整排序已经最优时不会切换。

当前分组违反硬门槛时，系统立即强制恢复到当时仍可路由的最低价分组。当前分组仍合格而另一个更便宜的分组出现或恢复时，这是普通 `Policy` 降价，必须通过第 8 节的统一防抖门槛。

## 5. 均衡模式：Deadline 优先

均衡模式始终使用显式预计输出量和 Deadline 做决策，不再存在倒计时或归零后转经济的分支。用户设置“预计输出 Token”和“用户容忍度”，其中预计输出量默认是 1,000 Token：

```text
完成时间 = TTFT / 1000 + 预计输出Token / 输出Token每秒
硬 Deadline = 26.73 秒
有效 Deadline = 26.73 秒 + 用户容忍度
```

默认用户容忍度为 5 秒。

### 连续调用

当当前分组存在且调用间隔不超过 30 秒时：

1. 当前分组在 26.73 秒硬 Deadline 内完成：绝不切换。
2. 当前分组超时：只从有效 Deadline 内可完成的候选中选择切换总成本最低者。
3. 如果没有候选满足有效 Deadline：退化为最快完成的替代节点，避免持续超时。

均衡 Deadline 的切换成本是：

```text
switchCost = EffectiveMultiplier * (30 * outputTokens / 1,000,000 + 2.0)
```

其中 `2.0 * EffectiveMultiplier` 是上下文未命中的固定罚金。倍率越低，输出成本和切换罚金都越低。

### 冷启动

当前分组为空、调用间隔无效，或间隔超过 30 秒时视为冷启动。冷启动不继承连续对话的切换罚金，只在有效 Deadline 内选择有效倍率最低的候选。

如果没有候选满足有效 Deadline，则与连续调用相同，退化为完成时间最短的替代节点，并在审计中标记为“最快降级”。

### 与任务时长的关系

短任务、中任务、长任务是独立设置，不参与均衡 Deadline。它只用于用户选择速度模式时保留的传统自适应剩余 Token 估算，不会从时钟推导或改变均衡输出预算。

## 6. 速度模式与自适应切换

速度模式在有效偏好仍为速度时遍历所有候选。候选已有 `1..19` 个本地性能样本时直接拒绝速度切换；样本数达到 20 后，才按以下任一条件判断。样本数为 0 或上游没有提供样本计数时，不触发这条本地样本门槛：

1. 新节点输出速度比当前节点快超过 20%，且新倍率不高于当前倍率的 110%。
2. 新节点的端到端预计完成时间快超过 30 秒，且倍率不高于当前倍率。

速度模式接受的候选依次按预计完成时间、输出速度、净节省、有效倍率和分组 ID 排序。这样不会因为只有首 Token 更快就忽略长输出场景的生成速度。

## 7. 速度模式的传统自适应成本与时长估计

只有用户选择速度模式时，传统自适应分支才使用任务时长和下列估算。有效偏好保持速度或被调用间隔降为均衡时会使用这些估算；降为经济时则直接进入严格最低价分支：

```text
完成时间 = TTFT + remainingTokens / outputTokensPerSecond
切换罚金 = 300,000 * newMultiplier * $5 / 1,000,000
净节省 = remainingTokens * (oldMultiplier - newMultiplier) * $30 / 1,000,000
         - 切换罚金
```

任务时长 `Short`、`Medium`、`Long` 并不表示 API 的真实上限，而是这个传统速度分支对剩余输出量的规划估计。Token 已考虑子代理造成的双倍工作量：

| 时长 | 非经济模式的保守估计 | 经济模式的乐观估计 | 期望完成时间 |
| --- | ---: | ---: | ---: |
| 短任务 | 0 | 156,960 | 1 小时 |
| 1-4 小时 | 156,960 | 627,840 | 2 小时 |
| 4 小时以上 | 627,840 | 3,767,040 | 6 小时 |

保守剩余量不超过 1,000 Token 时，系统拒绝切换。短任务在非经济偏好下也会拒绝切换，以避免为了很小的收益丢失上下文。

### 自适应均衡保护

当用户选择速度、但调用间隔把有效偏好降为均衡时，候选必须同时满足：

- 净节省大于切换罚金的 50%。
- 新完成时间低于任务期望时间，或变慢幅度小于旧完成时间的 10%。
- 有效倍率下降超过 5%。

### 调用间隔降为经济

当用户选择速度、但调用间隔把有效偏好降为经济时，实际路由直接使用严格最低价策略，见第 4 节，不再以净节省或 24 小时完成时间作为切换条件。

## 8. 候选推荐与实际写入

推荐列表会展示每个候选的接受状态、拒绝原因、净节省、预计完成时间和排名。实际写入前还会检查当前路由状态：

- `Initial`、当前分组违反硬门槛时的 `ForcedRecovery`，以及用户强制分组的 `ManualOverride` 立即执行，不受策略防抖阻塞。
- 所有普通 `Policy` 切换都要求已满足 30 秒驻留、已经记录至少 6 次已完成评估，并且状态中已经记录同一目标的 2 次稳定观测；Economy 降价、Balanced Deadline 切换、Balanced 最快回退和 Speed 切换都在其中。仅旧版 `route-state.json` 没有 `LastPolicySwitchAt` 防抖基线时允许一次迁移切换，之后立即使用完整门槛。
- 稳定目标计数检查的是本轮提案之前的状态：前两次同目标提案只记录观测，下一次仍提议同一目标时才切换，因此通常是第三次提案。
- 选中的所有 active Key 会一起更新到同一目标分组。
- 模拟只计算并展示，不会更新 Key。
- 写入失败或只完成部分 Key 更新时，路由状态会失去确定性，下一轮会重新核对。

### 实时目标预检

启用健康检查后，只有真实路由周期中至少一把业务 Key 将写到目标分组时才可能产生实时预检副作用。若精确匹配目标 `平台 + GroupId` 的成功记录仍在有效期内，直接复用；否则按以下顺序执行：

1. 将专用健康 Key 临时移动到目标分组，并确认 AIHub 返回的 Key ID 与 `GroupId`。
2. 发送流式 `ping`，必须收到第一个有内容的 Token 才算成功。
3. 在不可取消的 `finally` 路径用 `CancellationToken.None` 恢复原分组，并再次确认响应。
4. 恢复完成且预检成功后，才开始写业务 Key。

可恢复的目标失败会记录为失败观测，在本轮排除该分组，并使用本轮开始时的原始路由状态重新评估下一个目标。所有候选均失败，或健康 Key 是否恢复存在不确定性时，本轮不写任何业务 Key。模拟、无需改变任何业务 Key、或没有目标时不会移动健康 Key，也不会发送实时目标探测。

### 失败分类与响应解析

无 HTTP 状态的传输失败、HTTP 408/429/5xx，以及不属于全局配置问题的普通协议失败可以表示目标节点失败，因而允许记录失败并尝试本轮下一个候选。以下错误会中止整个路由周期，而且不会污染任何节点健康状态：本地配置错误、AIHub 账号或控制平面错误、调用方取消、上游 HTTP 401/403，以及经过安全归一化后为 `invalid_api_key`、`authentication_error`、`unauthorized`、`forbidden`、`model_not_found`、`invalid_model` 的全局错误码。

HTTP 200 也不等于业务成功。普通 JSON、`application/*+json` 和 SSE 数据都会检查业务错误；`success: false`、非空 `error`、失败 `code/status` 的优先级高于成功载荷。成功包装会解开 `data`、`result` 或 `payload`，同时仍支持直接业务对象；错误包装、缺失载荷、畸形 JSON/SSE 或没有内容 Token 的探测都不能成为成功。安全诊断不会回显响应正文、凭据形态的值，也不会把 HTTP 200 的业务错误显示成荒谬的“HTTP 200 请求失败”。

## 9. 何时重新计算

手动刷新、模拟、立即路由和自动轮询都会读取新数据并重新计算。供应商数据、账户可用分组和 Key 状态的缓存/刷新策略由路由服务负责；算法不会依据旧推荐直接写入。

## 10. 重要边界

- 这是分组路由，不是模型选择器。
- 无效倍率和未授权分组不会成为合格候选；没有 `CheckedAt`、新鲜健康成功、有效首 Token 延迟或有效输出速度中任一证据时同样会被排除。
- 经济、均衡和速度的显示权重不绕过 Deadline、上下文罚金、成功率、最新启用/可用状态或新鲜健康失败等硬规则。
- 所有金额均为估算值，实际账单由 AIHub 侧的计费与账户倍率决定。
